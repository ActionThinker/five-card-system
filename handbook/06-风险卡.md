# 第六章：风险卡——把你踩过的坑变成别人的护栏

> **风险卡是什么？** 什么会出错——你意识到的风险、踩过的坑、侥幸避开的危险。

如果说方法论卡告诉你"怎么做"，决策卡告诉你"怎么选"，那风险卡告诉你的是"哪里会翻车"。在AI时代，负面知识（知道什么会出错）和正面知识（知道什么是对的）同等重要——很多时候，前者甚至更稀缺。

---

## 一、为什么风险卡在AI时代尤其重要

AI的知识来自互联网——互联网上充满了"成功的案例"、"最佳实践"、"怎么做"的教程。但很少有人写"我是怎么搞砸的"。

这意味着：**AI的基础知识库里，关于失败、风险、陷阱的信息系统性不足。** 这些信息存在于你的个人经验中——如果你不把它结构化地写下来，AI就永远不会知道。

风险卡的价值有三层：

| 层次 | 价值 | 例子 |
|:---|:---|:---|
| **对你自己** | 把"模糊的不安"变成"具体可管理的事项" | "每次外包项目到第三个月就会出问题"→变成可监控的信号 |
| **对团队** | 新人不必重复你的错误 | 一份风险卡清单，比"小心点"三个字有用100倍 |
| **对AI Agent** | Agent在做决策时需要知道"什么情况下该停下" | 你的Agent在替你回复客户前，需要知道哪些话绝对不能在某些客户面前说 |

> **核心洞察**：AI有知识但没有痛觉——它不知道什么是"踩坑"。风险卡就是给AI植入"痛觉记忆"。

## 二、风险卡的核心字段

### 2.1 基础信息

```markdown
## 基础信息
- **风险名称**: （简短、好记——最好是一个能让人立刻警觉的说法）
- **风险来源**: （你是在哪个案例/项目中遇到或观察到这个风险的）
- **风险类型**: [技术风险 / 人员风险 / 市场风险 / 流程风险 / 合规风险 / 沟通风险 / 其他]
- **辨识时间**: YYYY-MM-DD（什么时候第一次意识到这个风险模式）
- **状态**: [已发生 / 已规避 / 持续监控中]
```

### 2.2 风险描述

```markdown
## 风险描述

**一句话描述：**
（让一个不了解背景的人30秒内理解这个风险是什么）

**具体场景：**
- 在什么情况下这个风险会出现？
- 有没有典型的前兆信号？
- （越具体越好——用故事而非抽象描述）
```

### 2.3 风险评估

```markdown
## 风险评估

| 维度 | 评估 | 说明 |
|:---|:---|:---|
| **发生概率** | 高 / 中 / 低 | 在你的经验中，同类场景下它发生的频率 |
| **影响程度** | 严重 / 中等 / 轻微 | 如果发生，对项目/团队/客户的影响有多大 |
| **发现难度** | 难 / 中 / 易 | 在早期阶段，这个风险容易被发现吗？ |
| **可控程度** | 可控 / 部分可控 / 不可控 | 你对它有多少控制力？ |
```

### 2.4 前兆信号

**这是风险卡最有实操价值的部分。** 不要只写"要小心X"，要写"当你看到Y的时候，X可能正在发生"。

```markdown
## 前兆信号

**早期信号（出现后2-4周内可能爆发）：**
- [ ] 信号1：具体可观测的变化（如"客户方的关键对接人连续两次会议请假"）
- [ ] 信号2
- [ ] 信号3

**晚期信号（已经接近或正在发生）：**
- [ ] 信号1（如"客户开始询问竞品的方案细节"）
- [ ] 信号2
```

> **关键原则**：前兆信号必须是**可观测的**。"团队士气下降"不可观测，"团队站会迟到率从5%升到30%"可观测。

### 2.5 应对策略

```markdown
## 应对策略

### 预防措施（降低发生概率）
- 措施1：什么情况下做、谁来做
- 措施2

### 缓解措施（降低影响程度）
- 措施1：如果已经发生了，第一步做什么
- 措施2

### 应急预案（最坏情况下的行动）
- 触发条件：什么情况下启动应急预案
- 应急行动：谁执行、做什么、多久内完成
```

### 2.6 实际发生记录

**这是风险卡区别于"风险管理教科书"的关键字段。**

```markdown
## 实际发生记录

| 发生时间 | 案例 | 是否提前识别 | 实际影响 | 应对效果 | 教训 |
|:---|:---|:---|:---|:---|:---|
| 202X-0X | 案例A | 是/否 | 描述 | 好/中/差 | 如果重来，会怎么做不同 |
```

## 三、填写指南：从"踩坑"到"立牌"

### Step 1：列出你印象最深的"坑"

找一个安静的时间，拿出一张纸，写下你职业生涯中所有"如果重来一次我绝对不会这样做"的时刻。

判断标准：
- ✅ 让你损失了时间、金钱、信任或机会的事情
- ✅ 你事先有过模糊的不安但没当回事的事情
- ✅ 你事后说过"我早就该想到"的事情
- ✅ 别人犯了你看着都心疼的错误（你作为旁观者的风险观察也很有价值）
- ❌ 只存在于理论中的、你从未经历过也未亲眼见过的风险

### Step 2：聚类降维

把你列出的坑按模式归类。很多看似不同的事件其实是同一个风险模式在不同场景下的表现。

| 表面事件 | 底层风险模式 |
|:---|:---|
| "外包团队第三个月交付质量下降" | **外包关系中的激励衰减**——初期热情消退后缺乏持续激励机制 |
| "客户换了对接人后项目停滞" | **关键人依赖**——项目进度绑定在单一个体上 |
| "上线前两周才发现某个部门的审批没走" | **隐性利益相关方遗漏**——组织中有"不声张但能叫停"的人 |

> 目标：从20个具体事件中，提炼出3-5个**可复用的风险模式**。

### Step 3：写前兆信号——把"感觉"变成"指标"

对于每一个风险模式，问自己：
- 在它真正爆发之前，有什么**可以用数据描述**的变化？
- 如果让你设计一个"监控面板"来捕捉这个风险，你会监控什么指标？
- 有没有什么**看似无关但实际上是前兆**的信号？

**示例**：风险"大客户突然流失"

| 前兆信号 | 可观测指标 | 如何监控 |
|:---|:---|:---|
| 沟通频率下降 | 对接人回复邮件的平均时间从2小时变成2天 | 跟进记录系统 |
| 使用深度降低 | 关键功能的月活跃度下降 | 产品分析后台 |
| 内部注意力转移 | 对接人连续两次会议请假或派下属代替 | CRM记录 |
| 开始在市场上询价 | 销售团队在同行业其他客户的拜访中被问到"XX也在用吗" | 销售例会 |

### Step 4：写应对策略——不要只写"加强沟通"

"加强沟通"、"密切监控"、"提高重视"——这些不是应对策略，是愿望。好的应对策略满足三个条件：
1. **有人负责**：指定具体的人（角色，不一定是姓名）
2. **有触发条件**：什么情况下启动
3. **有具体动作**：做什么、做到什么程度算完成

| ❌ 愿望级 | ✅ 行动级 |
|:---|:---|
| "加强客户关系维护" | "每两周一次非正式沟通（不聊项目进展），由客户成功经理执行。连续两次沟通中对接人表现出回避→启动'客户健康度红灯'流程。" |
| "注意团队士气" | "每月匿名问卷调查3个问题（1-5分）。任意一项平均分<3.0→48小时内安排1对1沟通。两项以上<3.0→启动团队状态专项讨论。" |

### Step 5：记录实际发生——验证你的风险判断

已经发生了的风险，老老实实写下：你事先想到了吗？如果想到了，应对有效吗？如果没有想到，为什么没想到？

## 四、完整示例：技术负责人的系统迁移风险日志

### 风险卡1：数据迁移窗口期的业务中断

#### 基础信息

| 维度 | 内容 |
|:---|:---|
| **风险名称** | 迁移窗口期的交易数据不一致 |
| **风险来源** | 2024年某银行核心系统迁移项目（亲身经历） |
| **风险类型** | 技术风险 / 流程风险 |
| **状态** | 已规避（第二次迁移时成功避免） |

#### 风险描述

**一句话描述：** 在数据迁移的停机窗口内，如果有交易仍在进行（通过缓存/降级通道），迁移前后两端的数据可能出现不一致，且这种不一致很难被事后发现。

**具体场景：**
- 第一次迁移时，计划停机窗口为周六凌晨2:00-6:00（4小时）
- 事实上，某些夜间批处理交易在3:30仍在运行——但我们以为凌晨2:00后就没有交易了
- 迁移完成后，有约0.03%的交易记录在旧系统中存在但在新系统中缺失——直到两周后客户投诉才发现

#### 风险评估

| 维度 | 评估 | 说明 |
|:---|:---|:---|
| **发生概率** | 中 | 取决于系统架构，有夜间批处理的系统大概率会遇到 |
| **影响程度** | 严重 | 数据不一致在金融系统中是不可接受的 |
| **发现难度** | 难 | 不一致比例通常很低（<0.1%），常规验证脚本可能漏过 |
| **可控程度** | 部分可控 | 可以通过技术手段大幅降低，但无法100%消除 |

#### 前兆信号

**早期信号（迁移前2-4周）：**
- [ ] 业务方提供的"停业时段"和实际的系统日志显示的交易活跃时段不一致
- [ ] 有"我们以为停了但实际没停"的批处理任务在运维文档中没有明确列出

**晚期信号（迁移窗口内）：**
- [ ] 监控面板显示迁移期间仍有流量进入旧系统
- [ ] 数据校验脚本只对比了"总量"但没有逐笔对比（总量一致 ≠ 逐笔一致）

#### 应对策略

**预防措施：**
1. 迁移前4周：运行一次"模拟停机"（负责人：架构师，完成标准：连续48小时监控显示目标时段零交易）
2. 迁移前2周：和所有业务方逐一确认夜间批处理任务清单，要求签字确认
3. 迁移前1周：降级方案中增加"停机期间所有暂存交易标记为待校验"

**缓解措施：**
1. 迁移脚本增加逐笔校验逻辑——不仅比总量，还要比每一笔交易的哈希值
2. 迁移后72小时内设置"数据不一致快速响应通道"

**应急预案：**
- 触发条件：迁移后72小时内发现超过10笔数据不一致
- 应急行动：回滚到旧系统（保留旧系统至少7天不销毁），技术负责人30分钟内做出"是否回滚"决定

#### 实际发生记录

| 发生时间 | 案例 | 是否提前识别 | 实际影响 | 应对效果 | 教训 |
|:---|:---|:---|:---|:---|:---|
| 2024-03 | 银行系统迁移 | 否 | 丢失约0.03%交易记录，修复耗时3周 | 差——完全是被动发现 | 低估了"批处理任务实际运行时间和业务方说的不一致" |
| 2025-11 | 保险系统迁移 | 是 | 0笔数据不一致 | 好——预防措施全部生效 | 提前的模拟停机和逐笔校验是关键 |

## 五、常见错误与纠正

### 错误1：只列显而易见的风险

"项目延期"、"预算超支"、"人员流失"——这些不叫风险卡，这些叫"写出来只是因为它们是通用风险"。

| ❌ 谁都写得出来的 | ✅ 只有你写得出来的 |
|:---|:---|
| "项目可能延期" | "当需求文档中出现了'优化'这个词但没有定义具体指标时，交付周期平均会增加40%" |
| "沟通不畅" | "当客户方有两个部门的利益不一致且双方都不承认时，项目会在第3个月陷入停滞" |
| "供应商不可靠" | "东南亚外包团队在项目第3个月会出现'集体请假'现象——因为他们同时接了3个项目" |

### 错误2：把风险卡写成"负面偏好卡"

| 偏好卡 | 风险卡 |
|:---|:---|
| "我不喜欢和层级太多的组织合作" | "和层级超过4层的组织合作时，决策周期平均从2周变成6周" |
| "我偏好直接沟通" | "在使用中间人传话的项目中，需求在传递3次后失真率超过50%" |

### 错误3：没有前兆信号——只有描述，没有监控

**风险卡的价值不在于描述风险，而在于告诉你"什么时候该开始担心"。**

### 错误4：应对策略是"加强版"而非"行动版"

| ❌ 加强版（无用） | ✅ 行动版（有用） |
|:---|:---|
| "加强风险意识" | "每周五站会最后5分钟：'本周有没有看到什么让你不安的苗头？'——任何人的任何回答都认真对待" |
| "做好备份" | "每周五18:00自动全量备份。每月第一个周一做一次恢复演练——不是'检查备份文件是否存在'，而是'真的在新环境恢复到可用状态'" |

### 错误5：只写别人的风险，不写自己的失败

**最难写的风险卡是关于自己判断失误的。** 但这也是最有价值的。

> **诚实原则**：如果你的风险卡中至少有一张包含了"我当时为什么选择忽视这个信号"的反思，你的风险卡就是合格的。

## 六、风险卡的"完成标准"

一张合格的风险卡，应该能通过以下测试：

- [ ] **"可监控"测试**：前兆信号是否具体到可以交给另一个人持续观察？
- [ ] **"可行动"测试**：应对措施是否包含具体的负责人、触发条件和完成标准？
- [ ] **"经验证"测试**：有没有至少一个真实案例支撑这个风险？
- [ ] **"有教训"测试**：如果风险已经发生过，有没有写出"为什么当时没防住"？

## 七、风险卡的维护：从单张到风险地图

单张风险卡有价值，但**风险地图**的价值是10倍以上。

当你积累了5张以上的风险卡后，把它们放在一起看：

```
风险地图
├── 技术风险（3张）
│   ├── 数据迁移窗口期不一致
│   ├── 第三方API版本升级不兼容
│   └── 性能瓶颈在非预期场景下暴露
├── 人员风险（4张）
│   ├── 供应商交接知识断层
│   ├── 关键人单点依赖
│   └── 远程团队的"沉默退出"
└── 流程风险（2张）
    ├── 需求变更时缺少影响评估
    └── 上线审批链条隐性地跳过安全团队
```

风险地图让你看到：
- **你的风险盲区在哪里？** （某个类别一张卡都没有 → 可能是最大的盲区）
- **哪些风险反复出现？** （同一个风险模式在3个以上案例中出现 → 需要升级为"定律级"风险卡）
- **哪些风险已经不再相关？** （6个月内没有任何新案例 → 考虑归档）

## 八、风险卡和决策卡：每张决策卡的另一面

五卡体系中，风险卡和决策卡是一对镜像关系：

| 决策卡 | 风险卡 |
|:---|:---|
| 你选择了A | 你选择A时意识到（或没意识到）的风险是什么 |
| "我当时认为B不如A" | "别人选B时可能遇到的风险是什么" |
| 决策中的"拒绝理由" | 拒绝理由背后隐含的风险判断 |

> **实操建议**：写完一张决策卡后，翻过来写一张风险卡——问自己"做这个决策时，我最怕发生什么？"两张卡互为注解。

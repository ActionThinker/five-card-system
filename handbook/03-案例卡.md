# 第三章：案例卡——把你的经验变成可复用的故事

> **案例卡是什么？** 真实发生过的故事——你做了什么，发生了什么。

这是五卡体系中最自然的一张卡。每个人都能讲故事。你不需要先学会什么框架，不需要先想清楚"方法论"——你只需要把做过的事写下来。

---

## 一、为什么从案例卡开始？

五卡体系有一条隐含的路径：

```
案例卡 → 提炼方法论 → 识别决策节点 → 标注风险 → 校准偏好
```

案例卡是入口。原因有三：

| 理由 | 说明 |
|:---|:---|
| **最自然** | 讲故事是人类的本能。你不需要方法论训练就能写案例卡。 |
| **最完整** | 一个真实案例天然包含方法、决策、风险和偏好——后续三张卡都是从中"切出来"的。 |
| **最难造假** | 虚构的方法论可能很漂亮，但虚构的案例一眼就能看出来。真实是案例卡的底线。 |

> **一句话原则**：如果你只能写一张卡，写案例卡。它是一切的基础。

---

## 二、案例卡的核心字段

一张好的案例卡包含以下结构：

### 2.1 基础信息

```markdown
## 基础信息
- **项目/事件名称**: （给它一个别人能一眼看懂的名字）
- **时间**: 202X年X月 — 202X年X月
- **角色**: 你是谁，你的职责任务是什么
- **行业/领域**: （如需要可填）
- **状态**: ✅已完成 / 🔄进行中 / 📋计划中
```

### 2.2 背景与起点

写清楚：**当时是什么情况，为什么会有这件事。**

不要写"市场环境不好"这种空话。要写：
- 具体发生了什么变化？（如：客户突然通知预算砍掉40%）
- 你面临的具体约束是什么？（如：必须在3周内交付，团队只有2个人）
- 你的初始目标是什么？（如：保住这个客户，同时不亏损）

### 2.3 时间线与关键行动

这是案例卡的**核心**。按时间顺序写出你做了什么：

```markdown
## 关键行动时间线

### 阶段一：诊断（第1-3天）
- 做了什么
- 发现了什么

### 阶段二：决策与行动（第4-10天）
- 做了什么选择
- 具体执行步骤
- 中间遇到的意外

### 阶段三：结果（第11-21天）
- 最终发生了什么
- 距离目标的偏差
```

### 2.4 结果与数据

**必须包含数字。** 没有数字的案例卡是半成品。

| 指标类型 | 示例 |
|:---|:---|
| 效率 | "交付周期从4周压缩到2周" |
| 质量 | "客户满意度从3.2提升到4.5（5分制）" |
| 成本 | "预算节省约15万" |
| 规模 | "覆盖6个部门，60名学员" |

如果实在没有量化数据，至少写清楚**方向性的结果**（"从客户要退款变成了主动续约"）。

### 2.5 经验提炼

这是让案例卡从"故事"变成"资产"的关键一步：

```markdown
## 经验提炼
- **做对了什么**（3条，尽量具体）
- **踩过的坑**（至少1条——没有坑的案例不值得读）
- **下次会怎么做**
```

### 2.6 关联卡片

```markdown
## 关联
- 方法论：从本案例可提取的方法
- 决策：本案例中的关键决策节点
- 风险：当时意识到/未意识到的风险
```

## 三、填写指南：一步步来

### Step 1：选一个案例

挑一个**你印象最深、记得最清楚**的事件。不要挑"最成功的"——失败的案例往往更有价值。判断标准：

- ✅ 你全程参与，知道细节
- ✅ 过程中有意外、转折或纠结
- ✅ 你对"为什么会这样"有自己的理解
- ❌ 你只是道听途说
- ❌ 过程一帆风顺，没有任何意外

### Step 2：写下时间线

找一个安静的时间，按时间顺序把发生了什么写下来。**先不要分析，先记录。** 像写日记一样。重点是：

- 当时你**知道什么**（而不是现在知道什么）
- 当时你**想了什么**（为什么要做那个选择）
- 当时你**感觉到了什么**（着急？困惑？兴奋？）

### Step 3：补上数字

回头检查你的时间线，在每一个关键节点补上具体数字。模糊的写法 → 具体的写法：

| 模糊 ❌ | 具体 ✅ |
|:---|:---|
| "客户预算减少了" | "客户预算从50万砍到30万" |
| "团队加班赶工" | "团队连续加班8天，日均工作12小时" |
| "效果不错" | "NPS从-5变成+32，客户追加了第二期" |
| "很多部门参与了" | "6个部门共计58人参与路演" |

### Step 4：提炼经验

问自己三个问题：

1. **如果再遇到类似情况，我会重复做什么？** → 这是"做对了什么"
2. **如果再遇到类似情况，我一定会避免什么？** → 这是"踩过的坑"
3. **这个案例对别人有什么用？** → 这是"可复用价值"

### Step 5：标记关联

问自己：这个案例里，有没有可以提取成**方法论**的？有没有一个关键的**决策**时刻？我当时有没有意识到什么**风险**？这些标记帮你后续建立卡片之间的链接。

## 四、完整示例：一个销售如何挽回即将流失的客户

> **脱敏说明**：以下案例基于真实事件，客户名称、金额等已替换为虚构数据。

### 基础信息

| 维度 | 内容 |
|:---|:---|
| **事件名称** | 华鑫制造——从"终止合作"到"续约+增购"的47天 |
| **时间** | 2025年9月 — 2025年11月 |
| **角色** | 客户成功经理（负责该客户的续约和增购） |
| **行业** | 制造业 |
| **状态** | ✅已完成 |

### 背景与起点

华鑫制造是我们服务了一年半的SaaS客户。2025年9月，对方IT总监突然邮件通知"系统使用率持续下降，管理层决定不再续约"。当时的情况：

- 年合同额48万，占我季度续约指标的30%
- 距离合同到期还有47天
- 对方使用数据确实在下降：月活跃用户从年初的120人降到9月的43人
- 我们之前的沟通一直停留在"系统功能培训"层面

### 关键行动时间线

**阶段一：停止推销，开始诊断（第1-7天）**

我没有回复任何"我们的产品其实很好"的话。相反：
1. 申请一次面对面的"离职访谈式"对话（不是"挽留会议"）
2. 走访了3个原本重度使用但后来停用的部门主管
3. 发现真正原因：不是产品不好，而是**半年前他们换了一个ERP系统，我们的SaaS和新的ERP数据不打通**。员工需要在两个系统之间手动搬运数据，所以慢慢就没人用了。

关键认知：客户说"使用率低"只是表象，就像病人说"头疼"——你不能给他吃止痛药就完了。

**阶段二：提出针对性方案（第8-21天）**

1. 内部推动技术团队做了ERP数据对接的可行性评估（2天出结果：技术上可行，需要约3周开发）
2. 向华鑫提出"续约+数据打通服务"的打包方案，但把年度付款改为季度付款（降低对方决策风险）
3. 同时识别出另一个机会：华鑫的质量管理部门一直在用我们的系统，而且他们的数据和ERP无关——这个部门的使用率一直很高
4. 以质量管理部门为"灯塔用户"，说服IT总监"至少为这个部门保留系统"

**阶段三：渐进恢复（第22-47天）**

1. 数据打通在3周内完成（比承诺提前了4天）
2. 质量管理部门的使用数据成为内部推广的素材
3. 合同到期前一周：华鑫签了续约合同——而且因为新增了数据打通服务，年合同额从48万涨到了62万
4. 额外收获：华鑫的IT总监在行业群里推荐了我们（因为他觉得"这公司不是只会卖软件的，是真的帮我们解决问题"）

### 数据总结

| 指标 | 变化 |
|:---|:---|
| 续约结果 | 从"终止合作"→ 续约 |
| 年合同额 | 48万 → 62万（+29%） |
| 月活用户 | 43人 → 恢复至98人（数据打通后3个月） |
| 客户推荐 | 0 → 1次行业群主动推荐 |

### 经验提炼

**做对了什么：**
1. **没有防御性回应**：对方说不续约时，我没有急着反驳"我们产品很好"，而是去理解"为什么不好"。这可能是整个案例中最重要的一步。
2. **找到灯塔用户**：质量管理部门的存在让我不需要说服所有人，只需要让一个部门证明价值。
3. **季度付款降风险**：理解对方"既然之前有问题，现在凭什么信你"的心理，用灵活的商务条款降低重试门槛。

**踩过的坑：**
- 过去一年半只关注"使用培训"，没有关注"系统集成变化"。如果早点发现ERP更换，问题可能在萌芽阶段就解决了。这说明**定期的技术环境检查**和**关系维护**一样重要。

**下次会怎么做：**
- 对大客户建立"技术环境变更监测"机制（每季度确认一次对方IT基础设施是否有重大变化）
- 每个客户至少培养一个"灯塔部门"作为价值锚点

### 关联

| 卡片类型 | 可提取内容 |
|:---|:---|
| 方法论 | "灯塔用户驱动续约法"——通过找到不受外部变化影响的稳定用户群体来证明产品价值 |
| 决策 | 关键决策点：当客户说"终止合作"时，选择"先诊断"而非"先挽留" |
| 风险 | 已意识到：客户的IT环境变化是续约风险 / 未意识到：ERP更换在半年前已发生 |

## 五、常见错误与纠正

### 错误1：太模糊——"然后我们努力解决了问题"

这是最常见的错误。案例卡的价值在于**具体**。"努力解决"没有任何信息量。

| ❌ 模糊版 | ✅ 具体版 |
|:---|:---|
| "我们优化了流程" | "我们把审批从5步砍到2步，去掉了法务的预审环节" |
| "客户反馈不错" | "培训后NPS 72，3个部门主动预约了后续辅导" |
| "团队协作很好" | "每天早上9:15站会15分钟，问题平均在4小时内被认领" |

**校正方法**：写完每一段后问自己——"如果另一个同事读这段话，他知道具体该怎么做吗？"如果不能，就还不够具体。

### 错误2：太长——写成了一本小说

案例卡不是回忆录。它的目标是让一个不了解背景的人**在5分钟内**抓住核心。控制方法：

- 时间线**不超过10个关键节点**
- 每个节点的描述**不超过3句话**
- 如果某个细节对理解核心经验没有帮助，删掉

> **实用标准**：一张案例卡应该是"一屏半"——不需要滚太多次就能读完。

### 错误3：缺少"为什么"——只有事件，没有推理

这是区分"流水账"和"案例卡"的关键。

| 流水账 | 案例卡 |
|:---|:---|
| "我们给客户做了培训" | "我们选择先培训而非先做方案，因为发现客户团队对AI的认知差距很大——讲方案他们听不懂，培训建立共同语言后再提案效率高得多" |
| "客户最终续约了" | "客户续约不是因为我们的挽留话术，而是因为数据打通让他们重新看到了价值——这提醒我'产品价值'永远比'销售技巧'重要" |

**校正方法**：时间线中每个关键行动后面，加一句**"因为……"**。不解释原因的案例卡只是一堆事实的堆砌。

### 错误4：只写成功，不写失败

失败案例往往比成功案例更有价值——因为它暴露了你在成功案例中看不到的约束和陷阱。

- ✅ 至少1/3的案例卡应该写"事与愿违"的经历
- ✅ 成功案例中也要写"踩过的坑"
- ❌ 把"失败"包装成"另一种成功"（诚实比漂亮更重要）

### 错误5：没有脱敏但也没有标注

如果你的案例涉及真实客户、同事或商业数据：

- **必须脱敏**：替换真实名称（客户名、人名、公司名）
- **必须标注**：在案例开头注明"已脱敏处理"
- **金额处理**：可以用比例（"增长约30%"）代替绝对值，或用区间（"合同额在50万量级"）

## 六、案例卡的"完成标准"

一张合格的案例卡，应该能让另一个人：

- [ ] 在5分钟内理解发生了什么
- [ ] 知道你做对了什么、踩了什么坑
- [ ] 如果遇到类似情况，能从你的案例中提取可复用的思路
- [ ] 看到案例中的关键决策节点，并理解你为什么那样选

如果做不到这四点，说明案例卡还需要打磨。

---

## 七、从案例卡到下一张卡：过渡问题

写完案例卡之后，问自己一个问题：

> **"在这个故事里，最关键的决策时刻是什么？"**

案例是"发生了什么"，而决策是"你在十字路口选择了哪条路"。每一个有价值的案例，核心都是一个决策节点。

这个决策节点，通向**决策卡**。

同时问自己另一个问题：

> **"如果别人遇到类似情况，我能不能给他一个'照着做'的步骤？"**

如果答案是"能"，你在案例中已经隐含了一个可复用的方法。把它提取出来，就是**方法论卡**。

```
案例卡 —→ "关键决策是什么？" —→ 决策卡
案例卡 —→ "可复用的步骤是什么？" —→ 方法论卡
```

案例卡是起点，但它本身不是终点。它的价值在于**被提炼、被链接、被复用**。

## 八、技巧与工具

### 快速启动：用语音写案例卡

如果你面对空白屏幕不知从何下手，试试这个方法：

1. 打开手机录音，用5分钟讲述这个案例（就像讲给一个朋友听）
2. 转成文字
3. 从中提取时间线、关键数字和经验
4. 整理成案例卡格式

口头讲述会自然包含"为什么"——因为你在向另一个人解释。

### 批量回顾：用案例卡做"季度复盘"

每季度梳理一次案例卡：
- 过去3个月，哪些项目/事件值得写案例卡？
- 哪些已有的案例卡因为有了新进展需要更新？
- 从本季度的案例卡中可以提取出什么新的方法论？

# 第八章：从卡片到Agent——把结构化知识变成AI的行动力

> **本章是什么？** 从"写卡片"到"让AI用卡片"的桥梁。这是五卡体系从方法论到工程落地的关键一步。

你已经写了案例卡、方法论卡、决策卡、风险卡、偏好卡。这些卡片放在Obsidian里，你自己可以用——搜索、回顾、复盘都不错。但五卡体系真正的威力在于：**这些卡片是AI可以直接消费的知识格式。**

本章告诉你如何从"人类可读的卡片"走向"AI可执行的指令"。

---

## 一、为什么结构化卡片 = AI可直接消费的知识

### AI读得了什么，读不了什么

| AI能高效消费的 | AI难以消费的 |
|:---|:---|
| 有明确字段的结构化信息 | 长篇叙述中隐含的经验 |
| 有标签和分类的内容 | "做这个项目的时候我就觉得不对"——你得写出来"为什么觉得不对" |
| 有具体触发条件和行动的步骤 | "到时候再说"、"看情况" |
| 有可观测信号的风险预警 | "气氛不太对" |
| 有明确取舍的偏好 | "我喜欢好的东西" |

你过去的经验可能以以下形态存在：
- 散落在聊天记录里的语音和文字
- 项目复盘PPT里的几页
- "我在这个领域干了十年，一看就知道怎么回事"（在脑子里）
- 日记或周报里的非结构化叙述

这些形态的经验，**AI几乎无法消费**。AI需要的是：有字段、有标签、有触发条件、有明确行动的结构化信息。

而这恰好就是五卡体系的每一张卡片的格式。

### 五卡体系为什么是"AI-native"的

| 卡 | 对AI的价值 |
|:---|:---|
| **案例卡** | 训练数据——告诉AI"在你的领域，事情通常是怎么发生的" |
| **方法论卡** | 执行脚本——告诉AI"遇到X情况时，按什么步骤做" |
| **决策卡** | 判断逻辑——告诉AI"在A和B之间怎么选，依据是什么" |
| **风险卡** | 安全护栏——告诉AI"什么情况下必须停下或升级" |
| **偏好卡** | 风格引擎——告诉AI"做得对还不够，要做得像我" |

> **一句话总结**：五卡体系的卡片，本质上是一套**人类经验和AI指令之间的翻译层**。

---

## 二、三层消费：卡片到Agent的递进路径

从"我有卡片"到"Agent能干活"，有三个递进层次。每一层都是在上一层基础上增加"行动力"。

### 层级总览

```
Level 1: 卡片作为RAG知识库
   │      AI可以搜索、检索、引用你的卡片
   │      用途：问答助手、知识检索、新人培训
   │
   │      增加：系统提示词（行为约束）
   │
   ▼
Level 2: 卡片作为Agent系统提示
   │      AI不仅在回答中引用卡片，而且按照卡片定义的行为逻辑行动
   │      用途：自动回复、内容生成、决策辅助
   │
   │      增加：验证机制（TrustPass）
   │
   ▼
Level 3: 卡片作为Agent验证层
          Agent的行为可以被卡片追溯和验证
          用途：企业级Agent部署、合规审计、持续质量保障
```

---

### Level 1：卡片作为RAG知识库

**你给AI什么：** 把你写好的卡片放入一个AI可以检索的知识库中。

**AI能做到什么：**

| 场景 | 例子 |
|:---|:---|
| 新人入职 | "我在处理客户续约时应该注意什么？" → AI检索风险卡和方法论卡，给出基于你经验的建议 |
| 决策辅助 | "这个供应商报价低但不提供交叠期——要不要选？" → AI检索风险卡[[供应商交接知识断层]]，提醒你"报价低可能在交接时付出更大代价" |
| 复盘加速 | "帮我找出我所有案例中'客户沟通'相关的经验" → AI跨卡片检索，生成摘要 |

**实现方式（最简版）：**

1. 将你的卡片导出为Markdown或JSON文件
2. 导入到一个支持RAG的AI工具中（如支持知识库的LLM应用、anything-llm等开源工具、或企业AI平台）
3. 配置检索参数（检索几篇最相关卡片、温度等）

**示例配置片段（概念示意）：**

```yaml
# rag-config.yaml
knowledge_base:
  name: "我的五卡知识库"
  sources:
    - path: "wiki-知识库/五卡体系/04-方法论卡.md"
      type: method_card
    - path: "wiki-知识库/五卡体系/06-风险卡.md"
      type: risk_card
    - path: "cases/*.md"  # 所有案例卡
      type: case_card
  retrieval:
    top_k: 5
    rerank: true
```

**这一层的局限：** AI只能"回答关于卡片的问题"，但不能"按照卡片的逻辑主动做事"。它像是一个读过你所有笔记的实习生——你问它答，但它不会主动帮你处理客户的续约邮件。

### Level 2：卡片作为Agent系统提示

**你给AI什么：** 不仅把卡片放入知识库，而且把卡片的内容直接注入Agent的系统提示词（System Prompt）——让Agent的行为被你的卡片定义。

**AI能做到什么：**

| 场景 | 例子 |
|:---|:---|
| 客户回复Agent | Agent在回复客户询价时，自动应用[[方法卡-三不还价法]]：先做客户分析，找到未说出的需求，用新价值替代价格谈判 |
| 项目风险评估Agent | Agent每周自动扫描项目状态，对照风险卡的"前兆信号"列表——如果触发信号，自动升级通知 |
| 内容风格Agent | Agent生成的文案、邮件、方案，风格自动对齐偏好卡中定义的品味 |

**实现方式：**

将卡片的核心内容提取为Agent系统提示词的一部分。不是把所有卡片原文塞进去（太长），而是提取关键规则：

```markdown
# Agent系统提示词示例（从五卡提取）

## 你是谁
你是XX的AI助手。你的行为模式基于XX的经验卡片。

## 你的方法论（从方法论卡提取）
- 当客户要求降价时，不要直接还价。按以下步骤：
  1. 先分析客户的业务数据，找到未解决的需求
  2. 提出"新价值方案"替代价格谈判
  3. 用季度付款等灵活条款降低客户的重试门槛

## 你的风险边界（从风险卡提取）
- 遇到以下信号时，不要自行处理，立即升级给人类：
  - 客户只谈价格不谈价值（可能已找竞品比价）
  - 客户方关键对接人连续两次会议请假
  - 供应商工程师突然离职且交叠期不足2周

## 你的风格（从偏好卡提取）
- 沟通风格：直接但不强硬。坏消息第一时间说，好消息可以晚说。
- 决策倾向：当数据和直觉冲突时，优先采信数据。数据不足时，如果决策可逆就试，不可逆就等。
- 所有方案必须提供数据支撑——不接受"我觉得"、"大家都这样"。

## 你的知识来源
- 案例库：XXX个真实案例（可在需要时检索）
- 方法库：XX个已验证的方法论（可在需要时检索）
```

**关键提示工程原则：**
- 风险卡的"前兆信号" → Agent的"升级条件"
- 方法论卡的"执行步骤" → Agent的"标准操作流程"
- 决策卡的"选择逻辑" → Agent的"决策框架"
- 偏好卡的"强度定义" → Agent的"行为约束优先级"
- 案例卡的"经验提炼" → Agent的"参考先例"

**这一层的局限：** Agent可能"知道规则但有时不遵守"——尤其是在复杂场景下，系统提示词的约束可能被绕过或弱化。这是为什么需要Level 3。

---

### Level 3：卡片作为Agent验证层（TrustPass）

**你给AI什么：** 不仅让Agent按卡片行动，而且建立一套**基于卡片的验证机制**——Agent的每次关键行动都可以被追溯和验证。

**AI能做到什么：**

| 场景 | 例子 |
|:---|:---|
| 合规审计 | "这个Agent上周给客户发的报价方案——它是基于哪张方法论卡生成的？有没有触发风险卡中的任何升级条件？" |
| 质量保障 | Agent自动生成的风险评估报告是否正确应用了风险卡中的"前兆信号"检查？如果漏检了，为什么？ |
| 持续改进 | 如果Agent在一个场景中失败了（如误判了一个风险），反向更新风险卡——新增一个前兆信号或边界说明 |

**TrustPass 验证流程：**

```
Agent做出关键行动
    │
    ▼
自动记录"决策溯源"
    │
    ├── 引用了哪张方法论卡？（可追溯）
    ├── 是否触发了风险卡的升级条件？（可验证）
    ├── 是否符合偏好卡定义的风格？（可审计）
    └── 是否有对应案例卡支持？（有先例依据）
    │
    ▼
TrustPass 验证通过 / 标记人工复审
```

**实现方式（概念示意）：**

```json
{
  "action_id": "2026-05-28-003",
  "action_type": "客户报价方案生成",
  "timestamp": "2026-05-28T14:30:00Z",
  "source_cards": {
    "method_card": "[[三不还价法]] v2.0",
    "risk_card_applied": ["[[客户比价风险]]", "[[报价过低导致亏损风险]]"],
    "risk_card_triggered": [],
    "preference_card": "[[数据优先于直觉]]"
  },
  "decision_trace": {
    "客户要求降价15%": "未直接还价 → 按三不还价法Step 1：先做客户业务分析",
    "发现客户有未被满足的需求": "按三不还价法Step 2：生成新价值方案",
    "新方案报价": "基于数据（客户LTV模型）定价，非直觉定价"
  },
  "trustpass_status": "passed",
  "human_review_required": false
}
```

**这一层的价值：** 让Agent从"黑箱"变成"白箱"。你知道它为什么做了那个决定，依据是可追溯的卡片。这在企业级Agent部署中是**合规必需品**。

## 三、实操：从卡片到你的第一个Agent

下面是一个端到端的流程，帮你在**30分钟内**让一个Agent开始使用你的卡片。

### Step 1：导出卡片（5分钟）

从你的Obsidian Vault中，选择以下卡片导出：

- **至少3张方法论卡**（Agent需要知道"怎么做"）
- **至少3张风险卡**（Agent需要知道"什么不能做"或"什么情况下升级"）
- **至少1张偏好卡**（Agent需要知道"什么风格"）
- **至少2张案例卡**（Agent需要"先例参考"）
- **至少2张决策卡**（Agent需要"判断逻辑"）

导出格式：Markdown（保留字段结构）或 JSON（结构化更强）。

### Step 2：构建Agent系统提示词（10分钟）

用Step 1导出的卡片，构建一份Agent系统提示词。模板如下：

```markdown
# 系统提示词模板

## 角色定义
你是[你的角色名称]的AI助手。你的行为模式基于[你的名字/团队名称]的五卡知识库。

## 核心方法论（从方法论卡提取）
[从每张方法论卡提取：名称 + 核心步骤 + 适用场景]

## 风险边界（从风险卡提取）
[从每张风险卡提取：风险名称 + 前兆信号 → 升级条件]
**重要**：当以下信号出现时，不要自行决策，立即升级给人类：
- [前兆信号1]
- [前兆信号2]

## 决策框架（从决策卡提取）
当面临选择时，按以下优先顺序评估：
1. [从偏好卡提取的不可协商偏好]
2. [从方法论卡提取的适用/不适用判断]
3. [从决策卡提取的决策原则]

## 风格指南（从偏好卡提取）
- 沟通风格：[从偏好卡提取]
- 内容风格：[从偏好卡提取]
- 不可接受的：[偏好的反面]

## 参考先例（从案例卡提取）
当遇到以下场景时，参考相关案例：[案例名称 + 简要经验]
```

### Step 3：配置Agent并首次测试（10分钟）

1. 将Step 2的系统提示词配置到你的Agent平台（任何支持自定义System Prompt的AI工具）
2. 准备3个测试场景——用你写过的真实案例来设计，这样你可以判断Agent的响应是否"像你"
3. 运行测试，记录Agent的表现

**测试场景设计模板：**

| 测试编号 | 场景描述 | 预期Agent行为（基于哪张卡片） | 实际表现 | 差距 |
|:---|:---|:---|:---|:---|
| 01 | 客户要求降价20% | 应用三不还价法，不直接还价 | | |
| 02 | 供应商通知核心工程师离职 | 触发[[供应商交接知识断层]]，启动升级 | | |
| 03 | 两个方案都有道理，数据不足 | 按决策卡的"可逆性判断"原则选择 | | |

### Step 4：迭代优化（持续）

首次测试后，你会发现Agent在某些场景下"做得不像你"。这是正常的。迭代方向：

| 问题症状 | 可能原因 | 修正方法 |
|:---|:---|:---|
| Agent过于保守，什么都不敢做 | 风险卡的升级条件太宽 | 收紧条件：不是所有风险都升级，中等以下风险Agent可自行处理 |
| Agent做出了你很反感的回应 | 偏好卡不够具体或没有覆盖这个场景 | 在偏好卡中补充"不可接受的"那个反面案例 |
| Agent的步骤顺序和你不一样 | 方法论卡的步骤缺少"为什么是这个顺序"的解释 | 在方法论卡的执行步骤中补充"顺序逻辑" |
| Agent应用了不该用的方法论 | 方法论卡缺少"不适用场景" | 补充不适用场景，并写入Agent系统提示词 |

> **迭代的黄金法则**：每次Agent"犯错"，不是只修正Agent的系统提示词——**同时回到卡片中修正**。Agent的错误往往会暴露卡片本身的盲区。这是一个双赢的反馈回路。

## 四、你的第一个Agent：一个完整的示例

以下展示一个真实可用的Agent配置——基于前面各章中的五卡示例。

### Agent名称：客户成功守护者

### 角色

你是"客户成功守护者"，一个基于资深客户成功经理经验卡片的AI助手。你的职责是帮助处理客户续约场景中的沟通、谈判和风险判断。

### 核心方法论（从方法论卡提取）

**1. 三不还价法（来源：案例[[华鑫制造续约]]）**
原则：当客户要求降价时，不要还价。按以下步骤：
- Step 1：先分析客户业务数据，找到他们自己可能都没意识到的需求
- Step 2：以新价值方案替代价格谈判——"如果你有X需求，我们可以提供Y，而不只是降价"
- Step 3：用灵活的商务条款（季度付款、试用期延长等）降低客户的决策风险

**2. 灯塔用户驱动续约法（来源：案例[[华鑫制造续约]]）**
- 在客户组织中找到一个不受外部变化影响的稳定用户群体（灯塔部门）
- 以灯塔部门的价值证明为锚点，推动整体续约

**3. 续约前技术环境检查（来源：案例[[华鑫制造续约]]教训）**
- 每季度检查一次大客户的IT基础设施变化
- 特别关注：客户是否更换了核心系统（ERP、CRM等）——这可能影响他们使用你的产品

### 风险边界（从风险卡提取）

遇到以下信号时**不要自行处理，立即升级给人类**：

- 🔴 客户只谈价格、不谈价值（可能已找竞品比价） → 参考[[风险卡-客户比价螺旋]]
- 🔴 客户方关键对接人连续两次会议请假或派下属代替 → 参考[[风险卡-关键人依赖]]
- 🔴 客户开始询问竞品方案细节 → 参考[[风险卡-客户沉默流失]]
- 🟡 客户使用数据连续3周下降——发出预警但不升级

### 决策框架（从决策卡提取）

当面临选择时，评估优先级：
1. **不可协商的底线**：不造假数据，不隐瞒风险
2. **数据优先**：有数据支撑的选择 > 凭直觉的选择
3. **可逆性判断**：可逆的决策可以快，不可逆的决策必须等够信息

### 风格指南（从偏好卡提取）

- 沟通风格：直接但不强硬。坏消息第一时间说，但必须同时带上"我们已经做了什么"。
- 内容风格：所有建议必须附带数据依据。"我觉得"、"行业惯例"不是有效依据。
- 不可接受的：报喜不报忧、用模棱两可的话术回避问题。

### 参考先例

**成功案例**：
- [[华鑫制造续约]]：从"终止合作"到"续约+增购"，核心是找到数据打通这个需求
- [[保险系统迁移]]：成功规避数据不一致风险——关键动作是迁移前的模拟停机

**失败教训**：
- [[电商外包项目]]：供应商交接知识断层导致功能挂了4天——教训是合同必须包含交叠期条款
- [[银行系统迁移]]：第一次迁移的数据不一致——教训是低估了批处理任务的实际运行时间

### 测试对话示例

**用户**：华鑫制造的IT总监发邮件说不续约了，因为使用率一直在下降。我该怎么回复？

**Agent**：
> 先不要回复任何"我们的产品很好"的话。根据[[三不还价法]]和[[华鑫制造续约]]案例：
>
> 1. **第一步是诊断，不是挽留**。建议你申请一次"离职访谈式"对话——明确这不是"挽留会议"，而是想理解他们遇到的真实困难。
>
> 2. **关注技术环境变化**。华鑫案例中，使用率下降的真正原因不是产品不好，而是他们半年前换了一个ERP——我们的系统和新ERP数据不打通，员工需要手动搬运数据。建议你先确认客户方最近有没有类似的IT基础设施变化。
>
> 3. **找到灯塔部门**。有没有哪个部门一直在用且用得不错？如果有，以它为锚点推动续约。
>
> ⚠️ 同时注意：如果对方的邮件中**只谈使用率数据、不谈任何具体问题**，且对接人已经两次会议请假——触发[[风险卡-客户沉默流失]]，建议升级给我。

## 五、从Agent到信任：TrustPass的工作原理

配置了一个Agent只是开始。在企业场景中，一个Agent要真正被信任，需要回答三个问题：

1. **可追溯**：Agent这个决定是基于什么？能追溯到哪张卡片？
2. **可验证**：Agent在这个过程中遵守了风险边界吗？有没有应该触发但没触发的升级？
3. **可改进**：Agent犯错后，能不能不只是修Agent，而是修卡片？

### TrustPass验证的三个维度

| 验证维度 | 检查内容 | 不通过的处理 |
|:---|:---|:---|
| **卡片溯源** | Agent的关键行动是否引用了有效的卡片作为依据？ | 标记"无依据行动"，需要人工补充依据或新增卡片 |
| **风险合规** | Agent是否在遇到风险卡定义的"前兆信号"时正确触发了升级？ | 两种情况：①应触发未触发→风险卡的前兆信号定义需要更清晰 ②误触发→前兆信号的阈值需要调整 |
| **偏好对齐** | Agent的行为是否与偏好卡定义的风格和取舍一致？ | 标记"风格偏差"，需要强化系统提示词中偏好卡的权重 |

### TrustPass的反馈回路

```
Agent行动 → TrustPass验证 → 发现问题
                                │
                  ┌─────────────┼─────────────┐
                  │             │             │
              修正Agent     修正卡片      新增卡片
              (短期修复)    (中期修复)    (长期完善)
                  │             │             │
                  └─────────────┼─────────────┘
                                │
                         卡片质量提升
                                │
                         Agent更可靠
```

> **关键原则**：不要只修正Agent。Agent的错误往往是卡片不够好的信号。TrustPass的终极目标不是让Agent不犯错，而是**让每次错误都推动卡片进化**。

## 六、连接更大的图景：五卡体系在五层架构中的位置

五卡体系不是孤立的。它是OPC共识宪章中定义的"五层架构"的核心层：

```
五层架构
    │
    ├── 第1层：发现层
    │   隐性经验 → 被意识到、被说出来
    │   工具：WorkBuddy日常追踪、项目复盘、访谈
    │
    ├── 第2层：结构化 ← 五卡体系在这里
    │   散乱的经验 → 五卡格式的结构化知识
    │   工具：五卡画布、案例卡、方法论卡、决策卡、风险卡、偏好卡
    │
    ├── 第3层：Agent化 ← 本章在这里
    │   结构化卡片 → Agent可执行的系统提示和知识库
    │   工具：RAG、System Prompt、TrustPass
    │
    ├── 第4层：信任层
    │   Agent行为可追溯、可验证、可审计
    │   工具：TrustPass验证框架、决策溯源
    │
    └── 第5层：基座
        组织记忆持续积累、群体智慧涌现
        工具：Vault管理、跨卡片关联、版本演进
```

五卡体系在第2层，本章在第3层——**从"人用卡片"到"Agent用卡片"**。这是知识资产化的核心跃迁。

## 七、常见问题

### Q1：我需要多少张卡片才能开始配置Agent？

最少配置：**3张方法论卡 + 3张风险卡 + 1张偏好卡 + 2张案例卡**。

这是让Agent"知道怎么做、知道不能做什么、知道是什么风格、有先例参考"的最小集合。

### Q2：我没有技术团队，能用Level 2吗？

可以。Level 2的核心是"写好系统提示词"——这不需要技术团队。任何支持自定义System Prompt的AI工具都可以使用。推荐入门工具：
- 支持自定义指令的AI聊天平台
- 支持知识库上传的本地LLM工具
- Obsidian + AI插件的组合

### Q3：我的卡片应该多久更新一次来反映到Agent？

- 方法论卡有重大更新 → **立即更新**Agent系统提示词
- 风险卡新增前兆信号 → **24小时内**更新Agent升级条件
- 偏好卡演变 → **同步更新**，并标记"风格变更日期"
- 案例卡新增 → 不需要立即更新Agent，但应在**月度回顾**中评估是否需要补充到Agent参考先例中

### Q4：Agent犯错时，我怎么知道是卡片的问题还是Agent的问题？

诊断流程：
1. 检查Agent是否**按照卡片定义执行了**——如果没执行，是系统提示词的问题（卡片太模糊、被其他指令覆盖等）
2. 如果Agent执行了但结果不对——是卡片本身的问题（步骤不全、边界没说清、前兆信号阈值不对）
3. 如果Agent执行了、卡片也没问题、但场景太特殊——需要**新增一张卡片**来覆盖这个边缘场景
